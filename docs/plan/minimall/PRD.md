## 1. 产品概述

### 1.1 产品定位

在 `charlotte_savanna` Django 项目下新建 `minimall` app，构建一个**电子微型商城**，覆盖买家购物全流程和管理员后台管理全流程。核心目标是**完整的电商业务闭环**，而非追求高并发或海量商品。

### 1.2 用户角色

| 角色 | 标识 | 描述 |
|------|------|------|
| 匿名用户 | 未登录 | 可浏览商品、搜索筛选、查看详情 |
| 买家 | 已登录，`is_staff=False` | 注册登录、购物车、下单、支付、查看历史订单 |
| 管理员 | 已登录，`is_staff=True` | 商品 CRUD、分类管理、订单处理（发货/退款） |

## 2. 功能需求

### 2.1 买家视角

#### F1 — 商品浏览

| 项目 | 内容 |
|------|------|
| **功能描述** | 买家可在首页浏览推荐商品，通过分类树导航、关键词搜索、价格区间筛选查找商品，点击进入商品详情页 |
| **输入** | 搜索关键词、分类选择、价格区间、排序方式 |
| **输出** | 分页商品列表（名称、价格、主图、分类），商品详情（名称、描述、全部图片、价格、库存、分类） |
| **业务规则** | 已下架商品（`is_active=False`）不在列表中展示；缺货商品展示"暂时缺货"，不可加购 |
| **缓存** | 分类树全站缓存（手动失效）；商品列表按参数 hash 缓存 5 min；商品详情按 slug 缓存 10 min |
| **验收标准** | AC1: 首页展示推荐商品和分类导航；AC2: 点击分类显示该分类及其子分类下的商品；AC3: 搜索"手机"仅返回名称/描述匹配的商品；AC4: 价格区间筛选正确过滤；AC5: 下架商品不出现在列表中 |

#### F2 — 用户注册登录

| 项目 | 内容 |
|------|------|
| **功能描述** | 买家可注册账号、登录、登出。注册时填写用户名、邮箱、密码。可设置支付密码（6 位数字），用于下单后模拟支付 |
| **输入** | username, email, password, payment_password（可选） |
| **输出** | 注册成功 / 登录成功建立 session |
| **业务规则** | 用户名和邮箱唯一；密码需 Django 默认验证器校验；支付密码非必填，首次支付时可补充设置；登录失败不区分"用户不存在"和"密码错误" |
| **验收标准** | AC1: 注册成功后可登录；AC2: 登录后访问个人信息 API 返回正确数据；AC3: 登出后购物车 API 返回 401 |

#### F3 — 购物车

| 项目 | 内容 |
|------|------|
| **功能描述** | 登录买家可将商品加入购物车、修改数量、移除商品、清空购物车。购物车数据持久化到数据库，跨设备登录保留 |
| **输入** | product_id, quantity |
| **输出** | 购物车商品列表（商品名、价格、数量、小计、总价） |
| **业务规则** | 首次加购自动创建 Cart；同一商品重复加购时数量累加（而非新增行）；数量不能超过库存，超过时自动截断为库存值并提示；购物车仅关联所属买家，其他买家不可访问 |
| **验收标准** | AC1: 加购后购物车角标更新；AC2: 同一商品重复加购数量增加而非多行；AC3: 修改数量为 0 等同于删除；AC4: 清空后返回空列表；AC5: 未登录用户访问返回 401 |

#### F4 — 下单

| 项目 | 内容 |
|------|------|
| **功能描述** | 买家从购物车结算生成订单。选择收货地址后确认下单，系统扣减库存并清空购物车中已下单商品 |
| **输入** | address_id（从已有地址中选择或新建） |
| **输出** | 订单确认页 → 订单创建成功（返回订单号和总金额） |
| **业务规则** | 下单时校验库存是否充足，不足时返回明确提示；库存扣减在数据库事务中完成（`select_for_update` 防超卖）；订单商品存储快照（名称、价格），后续商品改名/改价不影响历史订单；订单号格式：`YYYYMMDDHHMMSS{user_id}{random_4}` |
| **验收标准** | AC1: 购物车有货时可下单成功，返回订单号；AC2: 库存不足时下单失败并提示；AC3: 下单后对应库存减少；AC4: 订单中商品价格与下单时一致（快照）；AC5: 下单后购物车不再包含已下单商品 |

#### F5 — 模拟支付

| 项目 | 内容 |
|------|------|
| **功能描述** | 买家在订单详情页输入 6 位支付密码完成模拟支付。无真实资金流转，仅将订单状态从"待付款"变更为"已付款" |
| **输入** | payment_password（6 位数字） |
| **输出** | 支付成功 / 密码错误 / 密码未设置 |
| **业务规则** | 密码哈希存储（`make_password`），不可逆；密码错误最多 5 次，超过后当日锁定支付功能；已支付订单不可重复支付；仅订单所属买家可操作支付 |
| **验收标准** | AC1: 正确密码支付成功，订单状态变为"已付款"；AC2: 错误密码支付失败，不改变状态；AC3: 已支付订单再次请求支付返回错误；AC4: 非本人订单无法支付 |

#### F6 — 订单查看

| 项目 | 内容 |
|------|------|
| **功能描述** | 买家查看自己的历史订单列表和订单详情。列表按时间倒序，详情展示订单状态、商品明细、金额、物流状态 |
| **输入** | 无（列表）/ order_no（详情） |
| **输出** | 订单列表（分页）/ 订单详情（含状态时间线） |
| **业务规则** | 仅展示本人订单；取消/确认收货等操作在详情页完成 |
| **验收标准** | AC1: 列表仅显示本人订单；AC2: 详情页状态时间线完整；AC3: 无法通过 order_no 查看他人订单 |

#### F7 — 收货地址管理

| 项目 | 内容 |
|------|------|
| **功能描述** | 买家可增删改查收货地址。下单时可从已有地址选择或新建 |
| **输入** | receiver_name, phone, province, city, district, detail, is_default |
| **输出** | 地址列表 |
| **业务规则** | 每个用户可有多个地址，有且仅有一个默认地址；新建地址时若设为默认，取消其他地址的默认标记；被订单引用的地址不可删除 |
| **验收标准** | AC1: 新增地址成功；AC2: 设默认后旧默认取消；AC3: 删除非默认地址成功 |

### 2.2 管理员视角

#### F8 — 商品管理

| 项目 | 内容 |
|------|------|
| **功能描述** | 管理员通过 Django Admin 后台对商品进行增删改查，包括名称、描述、价格、库存、分类、图片、上架状态 |
| **输入** | 商品字段 + 图片上传 |
| **输出** | 商品列表（可搜索、按分类筛选）、编辑/新增表单 |
| **业务规则** | 仅 `is_staff=True` 用户可访问 Admin；商品删除为软删除或真删除（取决于是否有关联订单，有关联订单时仅下架）；商品变更后需清除对应缓存 |
| **验收标准** | AC1: Admin 中可新增商品含多张图片；AC2: 修改商品价格后详情页立即反映；AC3: 下架商品在买家端不可见；AC4: 非管理员无法访问 /admin/ |

#### F9 — 分类管理

| 项目 | 内容 |
|------|------|
| **功能描述** | 管理员在 Django Admin 中管理多层商品分类（增删改，支持父子层级） |
| **输入** | name, slug, parent（可选） |
| **输出** | 树形分类列表 |
| **业务规则** | 分类树通过 django-mptt 管理；删除有子分类的父分类时需处理子分类（或阻止删除）；分类变更时清除 `category_tree` 缓存 |
| **验收标准** | AC1: 可创建多级分类（如"电子产品 > 手机 > iPhone"）；AC2: Admin 中展示为树形结构；AC3: 分类变更后买家端分类导航更新 |

#### F10 — 订单管理

| 项目 | 内容 |
|------|------|
| **功能描述** | 管理员在 Django Admin 中查看全部订单，支持按状态筛选、搜索订单号，执行发货操作，处理退款 |
| **输入** | 筛选条件 / 发货操作 |
| **输出** | 订单列表、订单详情 |
| **业务规则** | 仅"已付款"状态可发货；仅"已发货"状态可退款；发货后订单状态变为"已发货"；自定义 Admin action 支持批量发货 |
| **验收标准** | AC1: Admin 中可查看全部订单；AC2: 按状态筛选正常；AC3: 发货操作后状态变"已发货"；AC4: 非已付款订单不可发货 |

## 3. 非功能需求

### 3.1 性能

| 指标 | 目标 |
|------|------|
| 商品列表 API 响应时间（缓存命中） | < 50ms |
| 商品列表 API 响应时间（缓存未命中） | < 200ms |
| 下单接口响应时间 | < 500ms |
| 并发下单（同一商品） | 无超卖，库存扣减准确 |

### 3.2 安全

- 支付密码哈希存储（`make_password`），不存明文
- Session 认证，CSRF 保护保持开启
- 买家只能操作自己的购物车、订单、地址
- 管理员权限通过 `is_staff` 校验
- API 输入校验（库存非负、价格非负、数量正整数）

### 3.3 可用性

- 注册/登录表单前端 + 后端双重校验
- 下单失败时明确提示原因（库存不足 / 地址缺失等）
- 支付密码错误时提示剩余次数

## 4. 订单状态流转

```
待付款 (pending)
  ├── 支付成功 → 已付款 (paid)
  └── 取消 / 超时 → 已取消 (cancelled)

已付款 (paid)
  ├── 管理员发货 → 已发货 (shipped)
  └── 申请退款 → 已退款 (refunded)

已发货 (shipped)
  ├── 确认收货 → 已收货 (received)
  └── 申请退款 → 已退款 (refunded)

已收货 (received)
  └── 自动完成 → 已完成 (completed)
```

## 6. 范围边界

### 6.1 本次包含

- 买家注册登录 / 商品浏览搜索 / 购物车 / 下单 / 模拟支付 / 订单查看 / 地址管理

- 短信/邮件通知

- 移动端适配
- 部署与 CI/CD
- 国际化（i18n）
- 前端构建工具链（Webpack/Vite）

## 7. 验收测试场景

### 场景 1：完整购物流程
1. 匿名用户访问首页 → 浏览商品列表 → 搜索"耳机" → 点进详情
2. 注册账号 → 登录
3. 加购 2 件商品 → 查看购物车 → 修改其中一件数量
4. 新增收货地址 → 下单 → 输入支付密码 → 支付成功
5. 查看订单 → 订单状态为"已付款"

### 场景 2：管理员后台
1. 管理员登录 /admin/
2. 新增"电子产品 > 音频设备"两级分类
3. 新增"蓝牙耳机"商品（2 张图片、价格 299、库存 50）
4. 买家端可见该商品
5. 管理员查看订单 → 对已付款订单执行发货
6. 买家端订单状态变为"已发货"
7. 买家确认收货 → 状态变为"已收货"

### 场景 3：边界条件
1. 库存为 3 的商品，用户 A 下单 2 件（成功），用户 B 同时下单 2 件（第 2 件失败，提示库存不足）
2. 已下架商品不在搜索结果中
3. 非本人订单无法查看和支付
4. 错误支付密码 5 次后锁定
